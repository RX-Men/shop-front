## ⚠️ Проблемы и сложности

### 1. Архитектура компонента

Изначально сделал иконку **хардкодом**, но позже понял, что компонент должен быть более гибким и переиспользуемым.

Сначала использовал:

```html
img="image"
```

Позже переделал на property binding:

```html
[img]="image"
```

Так как иконка потенциально может изменяться в зависимости от сценария использования компонента, тоже самое сделал с компонентом spin.

---

### 2. Code Review и стили

Первоначальная реализация стилей не соответствовала стандартам проекта.

После code review пришлось:

- переделать структуру CSS;
- привести стили к шаблону проекта;
- использовать принятые naming conventions;
- адаптироваться под существующую UI-систему.

Это помогло лучше понять важность **единых стандартов в команде**.

## 🧠 Решения, выводы и мысли

### Что понял

- `input()` в Angular Signals работает иначе, чем классические `@Input()`;
- важно заранее продумывать API компонента;
- code review — это не про "код плохой", а про соответствие архитектуре и стандартам команды;
- reusable-компоненты лучше проектировать гибкими, но не переусложнять.

### Что можно улучшить

В следующий раз сначала продумывать:

1. API компонента;
2. требования проекта по стилям;
3. naming conventions;
4. reusable ли компонент или нет.

## 🌿 Git Workflow

Во время работы использовал отдельную feature-ветку:

```text
feat/empty-component
```

### Коммиты

- `feat: add empty state component`
- `refactor: update empty component markup`
- `style: use neutral token for empty state`
- `revert: restore app shell files`

### Что понял по Git

Во время review получил замечание по поводу **merge commit** в feature-ветке.

Нужно изучить workflow через:

```bash
git rebase main
```

вместо:

```bash
git merge main
```

чтобы история коммитов оставалась чище и линейнее.
