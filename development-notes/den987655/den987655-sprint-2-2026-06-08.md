## ⚠️ Проблемы и сложности

### 1. Структура страницы `About`

Сначала страница получалась слишком завязанной на разметку и моковые данные прямо в компоненте.

Позже разделил ответственность:

- данные участников вынес в `about.constants.ts`;
- текстовую информацию о компании вынес в `about.json`;
- отображение одного участника вынес в отдельный `TeammateCard`;
- сама страница `AboutComponent` стала собирать готовые блоки.

Так стало проще читать компонент и менять контент без правок всей разметки.

### 2. Работа с карточкой участника

Карточка оказалась сложнее обычного UI-блока, потому что у нее есть:

- front/back стороны;
- эффект переворота;
- изображение;
- внешняя ссылка;
- промокод;
- длинное описание.

Пришлось отдельно продумать:

- ограничение текста;
- доступность через `focus-within`;
- размеры изображения;
- использование дизайн-токенов;
- корректное поведение ссылки на обратной стороне карточки.

### 3. Роутинг

Во втором спринте стало понятнее, как в проекте устроены маршруты.

Работал с:

```ts

```

и общим объектом:

```ts

```

Основная сложность была в том, чтобы не хардкодить пути в разных местах и подключать новые страницы так же, как это уже
принято в проекте.

### 4. Code Review и тесты

После review пришлось несколько раз возвращаться к задачам:

- поправить тесты для `AboutComponent`;
- поправить тесты для `NotFoundComponent`;

## 🧠 Решения, выводы и мысли

### Что понял

- lazy routes через `loadComponent` помогают держать страницы независимыми;
- общие константы маршрутов лучше использовать вместо строк в разных файлах;
- данные страницы лучше отделять от логики компонента;
- reusable-компонент должен иметь понятный API через inputs;
- `focus-within` важен для доступности интерактивных карточек;
- тесты часто ломаются не из-за большой ошибки, а из-за недостающих импортов или изменений в шаблоне;
- code review помогает увидеть, где компонент не совпадает с архитектурой проекта.

### Что можно улучшить

В следующий раз сначала продумывать:

1. где должны лежать данные страницы;
2. какие части лучше вынести в отдельные компоненты;
3. какие shared-компоненты уже есть в проекте;
4. какие импорты понадобятся для тестов;
5. как компонент будет вести себя с клавиатуры, а не только при наведении мышью.

## 🌿 Git Workflow

Во время работы использовал отдельные feature-ветки:

```text
feat/teammate-card
feat/about
feat/not-found
```

### Коммиты

- `feat: add card component`
- `feat: add teammate-card component`
- `fix: fixed styles`
- `fix: fixed teammate-card component`
- `fix: fixed teammate-card styles`
- `fix: fixed about list`
- `feat: about pages with routes.ts`
- `feat: about pages with banner and description`
- `feat: about pages with banner and description, add router-link`
- `fix: fixed pages witch AboutComponent`
- `fix: fixed test`
- `feat: add not-found pages witch routes.ts`
- `fix: fixed not-found.component`
- `fix: fixed router link and styles`
- `fix: fixed color in button`
- `chore: router link import`

### Что понял по Git

Во втором спринте стало понятнее, что история feature-ветки должна быть аккуратной и читаемой.

Также понял что Epic это отдельные страницы либо большие фичи, и от них нужно делать отдельные задачи, и научился
расставлять метки в задачах на канбан доске, и лучше понимать процесс разработки с GitHub Project.

Нужно внимательнее следить за:

- названиями коммитов;
- тем, какие файлы попадают в ветку;
- синхронизацией с `main`;
- тем, чтобы не заносить в PR лишние изменения;
- исправлениями после code review отдельными понятными коммитами.
  Также нужно продолжить практиковать:

```bash
git rebase main
```

чтобы история оставалась линейной и без лишних merge commit.
