# Sprint 4: Authentication, Cart & Architecture — 06.07.2026

## Что было сделано

Этот спринт оказался самым насыщенным с начала обучения. Если в предыдущих спринтах основной задачей было реализовать новую функциональность, то теперь пришлось много времени уделить архитектуре проекта, исправлению замечаний после code review и пониманию того, почему некоторые решения считаются хорошими, а некоторые — нет.

Продолжил работу над модулем авторизации. Добавил обработку состояний загрузки и серверных ошибок для страницы регистрации пользователя. Реализовал отображение ошибки при попытке регистрации уже существующего пользователя, связав ответ сервера с механизмом валидации Reactive Forms через setErrors(). Благодаря этому ошибка отображается как обычная ошибка поля, а не отдельным сообщением.

Переработал жизненный цикл подписок. Логику отключения состояния загрузки перенёс в finalize(), а также начал использовать takeUntilDestroyed(), чтобы избежать возможных утечек памяти при уничтожении компонентов. Это позволило сделать код чище и убрать дублирование.

Большая часть работы была связана с корзиной. Реализовал централизованное хранение информации о товарах внутри CartService. Добавил вычисляемую структуру itemsBySku, а также методы isInCart() и getQuantityBySku(). Теперь все компоненты приложения используют один источник данных о состоянии корзины, а не пытаются самостоятельно определять, находится ли товар внутри неё.

После этого обновил ProductCard, каталог, главную страницу, поиск и страницу детальной информации о товаре. Теперь кнопка автоматически меняет своё состояние после добавления товара в корзину, отображает "In cart" и блокирует повторное добавление одного и того же товара. При этом состояние синхронизируется между всеми страницами автоматически благодаря сигналам.

Отдельно пришлось разобраться с тем, как работать с SKU. Несмотря на то что в административной панели Commercetools SKU фактически является обязательным полем, GraphQL-сгенерированные типы продолжают считать его nullable. В итоге нормализация данных была перенесена в адаптеры, где товары без SKU просто отбрасываются, а дальнейшая работа приложения уже строится только с гарантированно валидными данными.

Продолжил работу с сервисом авторизации. Реализовал обработку различных сценариев авторизации и регистрации, улучшил обработку ошибок и постепенно начал избавляться от прямой зависимости AuthService от CommercetoolsService.

Самой большой задачей стал рефакторинг архитектуры после code review. По замечанию ревьюера начал переходить на использование InjectionToken и провайдеров. Создал интерфейсы AuthProvider и CartProvider, зарегистрировал их через useExisting и постепенно перевёл сервисы на работу через абстракции вместо конкретной реализации CommercetoolsService. Пока рефакторинг ещё не завершён полностью, но основная структура уже подготовлена.

Также занимался исправлением остальных замечаний code review: улучшал обработку ошибок, упрощал код, избавлялся от повторений, пересматривал организацию сервисов и пытался привести проект к более чистой архитектуре.

Параллельно начал глубже изучать уже написанный ранее код проекта. Несколько дней ушло только на то, чтобы разобраться, как именно устроена авторизация, каким образом происходит объединение анонимной корзины с корзиной авторизованного пользователя, зачем используются refresh token, token cache и почему последовательность вызовов внутри сервиса имеет значение.

## Проблемы

Главной сложностью этого спринта оказался уже не Angular как таковой, а понимание архитектуры приложения.

После получения code review стало понятно, что "код работает" далеко не всегда означает "код написан правильно". Некоторые замечания сначала вообще казались странными. Например, почему AuthService не должен напрямую зависеть от CartService, если именно во время авторизации происходит объединение корзин. Только после нескольких обсуждений стало понятно, что речь идёт о разделении ответственности и уменьшении связанности между сервисами.

Очень много времени заняло понимание Dependency Injection на более глубоком уровне. До этого момента использование inject() воспринималось как обычная возможность получить сервис. После знакомства с InjectionToken стало понятнее, зачем вообще нужны интерфейсы, провайдеры и почему сервисы лучше связывать через абстракции.

Отдельной проблемой стала работа с Commercetools SDK. Многие методы внутри сервисов выглядели как обычная цепочка вызовов, но на практике за каждым из них скрывались разные механизмы авторизации, переключения клиента, хранения токенов и создания API-клиента. Пришлось буквально разбирать сервис по строкам, чтобы понять последовательность выполнения.

Неожиданной проблемой оказались GraphQL-типы. Несмотря на реальные ограничения API, генератор продолжал считать некоторые поля nullable. Из-за этого несколько раз возникали ошибки типизации в местах, где логически null уже был невозможен. Пришлось разбираться, на каком уровне приложения правильнее устранять подобные несоответствия.

Во время реализации состояния корзины долго не получалось найти место, которое действительно должно быть источником истины. Сначала хотелось передавать состояние между компонентами через Input и Output, однако постепенно стало понятно, что правильнее хранить всю информацию внутри CartService и позволить компонентам только читать вычисляемое состояние.

Также пришлось довольно много времени потратить на разбор замечаний ревьюеров. Некоторые из них требовали не исправить одну строчку, а полностью изменить подход к построению архитектуры.

## Решения

- Добавил обработку loading и server error для регистрации пользователя.
- Реализовал отображение ошибки "пользователь уже существует" через Reactive Forms.
- Перенёс очистку loading в finalize().
- Добавил takeUntilDestroyed() для безопасной работы с подписками.
- Централизовал хранение состояния корзины внутри CartService.
- Реализовал itemsBySku, isInCart() и getQuantityBySku().
- Добавил отображение состояния "In cart" во всех местах приложения.
- Перенёс ответственность за определение состояния корзины в CartService.
- Начал переход на архитектуру с использованием InjectionToken и провайдеров.
- Создал AuthProvider и CartProvider для постепенного разделения зависимостей.
- Исправил часть замечаний code review без изменения поведения приложения.
- Перенёс проверки SKU на уровень адаптеров данных.

## Что я узнал

За этот спринт лучше понял не столько Angular, сколько общие принципы построения архитектуры.

Стало намного понятнее, зачем нужны абстракции поверх сервисов, почему Dependency Inversion считается одним из основных принципов проектирования и как Angular реализует это через InjectionToken.

Лучше разобрался с работой Signals. До этого они использовались в основном для хранения простого состояния компонентов, а теперь удалось увидеть, как computed может использоваться для построения общего состояния приложения, которым одновременно пользуются несколько независимых компонентов.

Гораздо лучше понял устройство сервисов авторизации и корзины. Теперь намного проще объяснить, зачем происходит переключение клиента после входа пользователя, почему важно сначала авторизоваться, а уже потом загружать корзину, и как работает объединение анонимной и пользовательской корзины.

Также впервые столкнулся с ситуацией, когда приходится исправлять не функциональные ошибки, а архитектурные замечания. Это оказался совершенно другой опыт по сравнению с обычной разработкой новых функций.
