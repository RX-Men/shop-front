# Sprint 2: Home Page, Layout System & Team Operations - 08.06.2026

## What was done

- **Home Page Delivery:** Fully built the **Home Page** with a complete responsive-adaptive layout. Implemented and integrated the hero block, promo section, product sections, banner usage, carousel-based content flow, shared logo, and shared footer. This sprint moved the storefront from a collection of isolated components toward a coherent, production-like landing page.
- **Responsive Layout System:** Continued shaping the visual foundation of the project around the custom responsive grid and container logic discussed with the team during Sprint 2. I used the agreed Bootstrap-inspired breakpoint strategy as a baseline, adapted it for our project needs, and kept validating the Home page behavior down to very small mobile widths.
- **Home Composition & Shared Product Presentation:** Expanded the `product-section` flow and tied it into real page composition instead of leaving it as a mock-only experiment. This included structuring the Home page content, integrating banners and promotional sections, and preparing shared product presentation patterns that can later be reused across the catalog and product pages.
- **Custom Carousel Engineering:** One of the most interesting technical tasks for me was implementing the `Carousel` without external libraries. Before building it, I reviewed Angular-friendly carousel solutions and was specifically looking for something lightweight in the spirit of Glide.js. In practice, I did not find a truly light option that suited our stack and constraints. The most obvious candidate was `Swiper.js`, but for our project it felt too heavy to pull in just for this use case. I also discussed the decision with our mentor, and we agreed that the best direction was to build the carousel natively. As a result, I worked through autoplay behavior, responsive scaling, content flow, and adapting the slider to the overall Home page composition myself.
- **Shared Footer Delivery:** Created a full shared `Footer` component with structured JSON-driven navigation data, dedicated child navigation columns, and test coverage. This turned the footer into a reusable layout block rather than a one-off page fragment.
- **Shared Logo Creation:** Implemented the shared `Logo` component and spent much more time than expected preparing the actual logo asset. At first I thought logo generation would be quick, but in practice I had to iterate across several tools and workflows. I generated prompts with Gemini because image generation has a lot of prompt-specific nuances, experimented in Leonardo.ai with the NanoBanana model, and then finalized the vector-ready result through [Vectorizer.AI](https://vectorizer.ai/) because it was the practical option that could produce SVG output for the project.
- **Layout & Shared Component Expansion:** Continued building the shared UI layer and layout primitives used by the storefront. During the sprint I delivered or finalized work around `Banner`, `Carousel`, `Tooltip`, `Logo`, and `Footer`, and also prepared the next wave of UI/layout work in the branches `feat/ui-kit-header`, `feat/ui-kit-popover`, `feat/tooltip-enhancement`, and `fix/icon-placement-positioning`.
- **Header & Interaction Improvements:** Designed and implemented a reusable `Header` with a `SearchWidget`, `CartButton`, improved product card placement, autofocus behavior, and responsive Home page refinements. I also extended tooltip positioning logic and fixed re-rendering issues for dynamic tooltip content.
- **UI Architecture Cleanup:** During the header and layout work, I pushed the codebase toward better shared component boundaries. In particular, I moved product card logic into the shared layer, prepared supporting constants and types, and added a reusable autofocus directive to improve the header search flow.
- **Support for Adjacent Features:** Besides the main Home page scope, I also handled several smaller but important integration tasks around the repository structure, including content relocation and route corrections for the Sign Up flow right before the sprint deadline.
- **Team Lead / Manager Responsibilities:** Went much deeper into the Team Lead and manager role. Added an **Epic** column in GitHub Projects to track feature-level progress, created labels and task types, split the backlog into sprints, maintained meeting-note links, and prepared progress charts that I plan to present during the final project defense.
- **Board Governance & Delivery Control:** Formalized how large scopes should be decomposed into atomic tasks, clarified Definition of Done boundaries for page-level work, aligned the team on Kanban column progression, and monitored GitHub Projects Insights so the board could reflect real delivery velocity instead of vague status reporting.
- **Git Workflow Enforcement:** Spent additional effort aligning the team on clean git history practices. I repeatedly pushed the team toward `git rebase` over uncontrolled merges, explained why review fixes must be handled quickly, and generally took tighter control over how changes reach `main` so the repository history stays readable.
- **Git History Recovery Work:** Part of my time also went into correcting the consequences of teammates using merges in places where a cleaner history was expected. There was a real risk of breaking the repository's linear git history, but I managed to straighten it out and keep the branch flow under control.
- **Code Review Contribution:** Although my own branches were often reviewed very slowly, I consistently reviewed teammates' Pull Requests quickly and in large volume. There were moments when I reviewed up to **5 PRs in a row**, trying to keep team throughput moving and reduce waiting time in the review queue.
- **Technical Coordination & Interview Preparation:** Organized and led multiple synchronization meetings around Sprint scope, review bottlenecks, architecture expectations, Angular performance practices, Git workflow, and external mentor interview readiness. A meaningful part of my sprint was not only writing code, but also making sure the team understood why certain technical and process decisions mattered.
- **Design Direction & Frontend Standards:** Since the project moved forward without a dedicated in-team designer, I also took responsibility for aligning the visual direction of the Home page, layout behavior, typography decisions, and general component consistency while still trying to keep implementation grounded in accessibility and maintainable shared patterns.

## Problems

- **Missed Sprint-1 Follow-ups:** I did not finish the planned **commercetools Integration** and **State Management Architecture** tasks from my first diary entry. A large part of my time shifted into building the full Home page and covering team coordination gaps.
- **Slow Code Review Cycle:** My pull requests and commits are often reviewed too slowly, sometimes with a delay of **5-7 days**. This creates long feedback loops, slows down delivery, and forces me to context-switch back into old changes instead of moving forward.
- **Uneven Team Velocity:** Some tasks that should have been relatively straightforward, such as static `404` and `About` pages, were moving for up to two weeks. The situation improved, but delivery speed across the team still remained uneven. Because of this, I often had to dive into tasks directly, spend time on 1-to-1 syncs, and guide execution in detail instead of focusing on my own feature work.
- **High Coordination Overhead:** One of the biggest bottlenecks this sprint was the amount of follow-up needed to keep certain tasks moving. Without regular check-ins, task ownership could become fragile, and the Team Lead role gradually shifted from strategic coordination toward continuous operational support.
- **Capacity Planning Risk:** Looking ahead, I have some concerns about whether the team can finish the future `Cart Page` scope at the current pace. Based on the delivery speed I observed on simpler page tasks, I may need to rethink resource allocation and decide whether this feature should stay with its current owner or be partially reassigned to me or to another teammate so that we can complete the end-to-end user flow on time.
- **Lack of Designer & Mature Design System:** It is still difficult to move fast without a dedicated designer and without a fully established design system. In practice, this means many visual and UX decisions still need to be shaped directly in development. Because of this, I regularly had to invent UI solutions myself and adjust them on the fly when something did not look convincing in the real layout.
- **Design Validation Gaps:** Before the final stylistic corrections, I showed our Home page to a graphic designer acquaintance to get an external opinion. He pointed out several inaccuracies and a few critical design issues that I then tried to fix during the Home page polish pass. In particular, he did not like the generated logo very much, so I already know that I will most likely redesign it later. If possible, I would also like to show the implementation to UX/UI designer friends in the future to catch the most critical visual problems earlier.
- **UI Kit Complexity From Scratch:** Building a UI kit from zero is much harder than it may seem at first. Components like tooltips and popovers require thinking through many states and edge cases upfront, and at the same time I try to keep everything aligned with accessibility expectations by relying on W3C documentation and established UI pattern guidance.

## Solutions

- **Communication Reset:** We addressed the issue of teammates disappearing from the workflow. I raised it twice during team calls, and the situation improved: now people warn the team earlier if something is going wrong or if they are blocked.
- **Management Through Structure:** To compensate for weak execution visibility, I improved the GitHub Projects workflow with epics, labels, task types, sprint grouping, and visual tracking. This made it easier to see where progress was stuck and where intervention was needed.
- **Focus Reprioritization:** Instead of forcing too many architecture tasks in parallel, I concentrated on shipping the Home Page and strengthening the shared layout system first. This gave the project a visible user-facing result and a stronger UI foundation for the next sprint.
- **Hands-on Coordination:** When progress slowed down, I used direct check-ins and 1-to-1 communication to unblock people and keep work moving. This is not the ideal long-term model, but it helped prevent the sprint from stalling completely.
- **Task Sizing by Delivery Capacity:** I also realized that some workstreams currently move better when tasks are decomposed into smaller, clearly defined units with explicit expectations. Going forward, this means I will likely need to assign narrower tasks with more structured guidance, while keeping larger user-flow-critical features under closer delivery control.
- **Accessibility-Driven Implementation:** Even while making design decisions on the fly, I tried to keep implementation grounded in accessibility best practices by leaning on W3C documentation and established UI patterns instead of relying only on visual intuition.

## What I learned

- **Responsive Composition at Page Level:** Building the full Home Page taught me much more than isolated UI-kit tasks. I gained deeper experience in combining shared components into a coherent, responsive page with real content structure and reusable layout patterns.
- **Leadership Requires Systems, Not Only Coding:** Team productivity depends heavily on visibility and process design. GitHub Projects became much more useful after introducing epics, clearer labels, task types, and sprint framing instead of treating the board as a flat list of tickets.
- **Review Latency Damages Momentum:** Even when implementation quality is solid, slow reviews can seriously reduce sprint velocity. I felt how much engineering throughput depends not only on code, but also on a healthy review turnaround.
- **Autonomy Requires Deliberate Team Development:** I learned that a team does not become self-managed automatically. If expectations, ownership boundaries, and delivery habits are not reinforced consistently, the Team Lead can become a bottleneck very quickly and lose time that should have gone into feature delivery and architectural work.
- **UI Kit Work Is Real Product Design Work:** Building primitives like `Tooltip`, `Popover`, `Logo`, `Carousel`, and layout components from scratch is not just coding. It requires balancing architecture, visual consistency, accessibility, responsiveness, and future reusability at the same time.
