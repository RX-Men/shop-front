## What was done

- **Sprint 3 Plans Completion:** By the time of this entry, I managed to close the scope I had planned after Sprint 3. The catalog moved from a mock-oriented page shell to a real storefront flow with filters, sorting, pagination, product details, search, Home-page collections, and commercetools GraphQL integration.
- **Catalog Filters & Sorting Delivery:** Finished the catalog sidebar and toolbar work that had been in progress at the end of the previous sprint. Built the `Select`, `CheckboxGroup`, and `Drawer` shared components, connected them to the catalog filters UI, and made the desktop sidebar and mobile drawer work through the same controlled state model.
- **Reusable Drawer & UI State Coordination:** Extracted a shared `Drawer` component and introduced `UiService` so layout-level overlays do not conflict with each other. This made the search drawer and catalog filters drawer behave consistently, including active drawer tracking and body scroll locking.
- **commercetools Configuration Hardening:** Refactored commercetools configuration into a centralized `InjectionToken` and fixed the undefined environment case that appeared during builds. This made the API setup less fragile and easier to reuse across services.
- **API Research Path: HTTP to GraphQL:** I initially approached commercetools through regular HTTP requests. This helped me understand the platform structure, but the responses were very large and contained many fields that were not needed by any page in our application. While exploring the platform further, I discovered that commercetools also supports GraphQL and decided to switch the product-facing integration to GraphQL queries.
- **GraphQL Code Generation:** Added GraphQL codegen and generated commercetools types for product queries. This shifted integration work away from manually typed query responses and gave services/adapters a stronger contract with the API.
- **Type Generation Cleanup:** After getting advice from Gemini, I experimented with a broader GraphQL tooling setup that included Apollo-related packages. In practice, Apollo was unnecessary for our current integration style, so I removed the extra dependencies and kept only the packages needed directly for GraphQL type generation: `@graphql-codegen/cli` and `@graphql-codegen/typescript-operations`.
- **Live Catalog State:** Connected the catalog page to commercetools GraphQL and implemented real product fetching, filter data loading, sorting, pagination, and catalog state handling. This was the main step that moved the project from static storefront UI toward an actual e-commerce application.
- **Pagination Component & Catalog Pagination:** Built a reusable `Pagination` component and then integrated it into the catalog flow. The catalog now supports page navigation through the API instead of pretending that all products live in one local array.
- **Product Details Flow:** Implemented product details routing and layout, then connected the page to commercetools GraphQL. Added a dedicated service, adapter, constants, content JSON, and UI structure so a user can open a product from the catalog and see a real product detail page.
- **Layout Architecture Refactoring:** Moved `Header` and `Footer` from the shared component layer into `core/layout`, because they are application shell components rather than reusable UI primitives. This made the architecture cleaner before more feature work landed on top of it.
- **Search API Integration:** Connected the existing search drawer to commercetools GraphQL through a dedicated `SearchService`. Search stopped being only a UI interaction and became part of the real product discovery flow.
- **Home Page API Integration:** Replaced mock-driven Home product lists with commercetools-backed collections for new and trending products. This kept the Home page visually close to the previous implementation while making its data source real.
- **Product GraphQL Contract Synchronization:** Unified product GraphQL contracts across catalog, Home, search, and product details. Introduced product adapters and shared product mapping logic so UI components do not need to know about raw commercetools response shapes.
- **Discount & Price Presentation:** Added a reusable `Price` component and adapted product card/product detail UI to support discounted prices. This made product presentation closer to a real store instead of a flat mock catalog.
- **Review & Integration Control:** Continued merging and reviewing team work while keeping the branch flow under control. In practice, a large part of the sprint again included not only implementation, but also making sure features land in a coherent order and do not break the application structure.

## Problems

- **Procrastination Before commercetools:** As usual, part of the sprint time went into procrastination before starting the hardest unknown area. I felt tension before diving into the commercetools API because it required focused energy and learning something new. Once I actually started, the tool turned out to be interesting and I generally liked working with it, but the entry barrier still cost me time.
- **Oversized REST Responses:** The first HTTP-based commercetools requests returned huge response bodies with many fields that were irrelevant for our UI. This felt wrong for a frontend application because it increases payload size, slows down requests, and makes the client device process data it does not need.
- **GraphQL Learning Curve:** I had not worked with GraphQL before this sprint, so the syntax and mental model were not obvious at first. It took time to understand how to describe exactly what fields I need, how to navigate the commercetools schema, and how to translate storefront UI requirements into useful queries.
- **Endpoint Discovery & Permissions:** commercetools has many endpoints and query areas, and not all of them fit a customer-facing storefront. Some APIs I initially investigated were meant more for manager/admin scenarios rather than normal users. I also ran into missing permissions for the client ID and had to reissue credentials twice with broader access rights.
- **Deprecated or Inefficient API Areas:** While researching product data access, I found that some tempting options, such as `ProductProjections`, are considered outdated and slower. This forced me to spend more time checking documentation and choosing more appropriate GraphQL paths instead of taking the first endpoint that seemed to return product data.

- **Very Slow Team Velocity:** The delivery pace from the rest of the team remained extremely slow. Some scopes of roughly **150-400 lines of code** could be implemented and reviewed for about two weeks. The bigger issue is not only the raw speed, but also the lack of initiative, product thinking, and independent ideas from teammates.

## Solutions

- **Closed the Core Product Flow First:** I focused on the most important storefront path: Home, catalog, filtering/sorting, pagination, product details, search, and product presentation. This gave the project a coherent user flow even though some team-owned areas are still unfinished.
- **Switched to GraphQL for Data Shape Control:** Instead of continuing with large REST responses, I moved product fetching to GraphQL so each page can request only the fields it actually needs. This reduces payload size, makes responses easier to work with, and should be friendlier to the client device.
- **Learned Through Multiple Sources:** To get comfortable with GraphQL and commercetools, I used the commercetools interactive query tool, official documentation, Gemini explanations, a Visual Studio Code GraphQL extension, and a lot of practical trial and error. This combination helped me move from random exploration toward more deliberate query design.
- **Kept Integration Work Adapter-Based:** To avoid spreading commercetools response shapes across the UI, I introduced and refined adapters for product data. This made it possible to change GraphQL queries and discounts handling without rewriting components everywhere.
- **Reduced Tooling to What Was Actually Needed:** After testing a heavier setup with Apollo-related packages, I removed what the project did not use and kept the GraphQL codegen setup focused on type generation only. This kept the dependency surface smaller while still giving us typed operations.
- **Separated Layout From Shared UI:** Moving app shell pieces into `core/layout` helped clarify ownership boundaries. Shared components can stay focused on reusable UI primitives, while header/footer remain part of the application structure.
- **Accepted Scope Reality:** I stopped treating every missing area as something I must personally fix. The project needs authorization and profile work, but taking it over at this stage would turn the final sprint into pure exhaustion instead of meaningful learning.
- **Prioritized Presentation-Ready Flows:** Since the team presentation is close, I shifted attention toward flows that can be demonstrated clearly: catalog browsing, product discovery, details, prices, discounts, and API-backed data.

## What I learned

- **API Integration Changes the Whole Frontend Shape:** Once real commercetools data entered the application, page components, services, adapters, test data, product cards, search, Home sections, and details pages all had to align around one product contract.
- **GraphQL Became the Main Technical Discovery of the Sprint:** This was the first time I worked with GraphQL in a real feature, and I am genuinely glad and proud that I finally learned it. I especially liked the idea that the frontend explicitly defines the response shape instead of receiving a large generic payload and filtering it manually afterward.
- **Precise Queries Improve Product Thinking:** GraphQL made me think more carefully about what each page actually needs: catalog cards, search results, Home sliders, and product details all have different data requirements. This made the API layer feel closer to UI design rather than just "fetch everything and sort it out later".
- **GraphQL Codegen Is Worth the Setup Cost:** Code generation added setup complexity, but it gave much better confidence when working with commercetools responses and helped reduce fragile manual typing.
- **Tooling Advice Still Needs Verification:** Gemini helped me discover GraphQL code generation, but its first dependency suggestion was too broad for this project. I learned to treat AI advice as a starting point, test what is really needed, and then remove packages that do not serve the implementation.
- **commercetools Requires Careful API Reading:** The platform is powerful, but it is easy to choose the wrong endpoint or query area at first. Some APIs are legacy, slow, or intended for internal management use cases rather than customer-facing storefront pages.
- **Adapters Protect UI Components:** Mapping API responses into project-specific view models is not just cleanliness. It is what keeps product cards, price rendering, details pages, and search results from becoming coupled to backend response details.
- **Team Leadership Has a Limit:** Process structure, reviews, and task decomposition help only when people actually participate. If teammates lack initiative, the Team Lead can compensate for a while, but eventually the project turns into solo delivery.
- **Motivation Depends on Process Clarity:** Even when the code work is interesting, unclear final expectations, late process changes, and weak organizer involvement make it much harder to stay emotionally invested.
- **Finishing a Project Is Not the Same as Enjoying It:** I am glad the core product flow is mostly there, but the amount of unpaid coordination, review pressure, and missing team ownership made the result feel more like survival than a normal team project.
