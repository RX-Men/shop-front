# Sprint 1: Input Component - 25.05.2026

## Что было сделано

Реализован базовый InputComponent для UI-кита интернет-магазина. Компонент поддерживает двустороннее связывание через model(), состояния ошибки и предупреждения, disabled/readonly режимы, лейбл, подсказку, required-маркер, a11y-атрибуты (aria-invalid, aria-describedby). Написаны юнит-тесты, компонент покрыт data-testid через централизованный APP_TEST_IDS.

Мы строим свой UI-кит с нуля, и я взялся за свой InputComponent. Попутно накидал демо-страницу, чтобы ребятам из команды было удобнее смотреть как это всё выглядит вживую. Зашло — теперь из этого делают отдельную фичу, будет свой внутренний UI Kit Showcase.

## Проблемы

Правильный вопрос - что не было проблемой. Главной проблемой оказалось что при изучении **нового** фреймворка надо брать компонент попроще.

CI падал локально зелёный код. В Linux CI вылезал TS2300: Duplicate identifier 'provideRouter' — конфликт между app.spec.ts и app.config.ts. Локально на macOS не воспроизводилось.

Сначала я начитался про CVA и подумал что без него это и не инпут будет, а в итоге CVA оказался избыточным. Реализовал полный ControlValueAccessor, но колбэки через signal() давали неудобный синтаксис двойного вызова: this.onChange()(v). Потом выяснилось, что ReactiveFormsModule в проекте вообще не используется (so far). Короче компонент претерпел изменения в пользу упрощения.

Дизайн-токены пришлось переосмыслить дважды. Сначала создал семантический слой form-tokens.scss, а мне говорят - overengineering! Затем добавил градации --error-300/500/700 по аналогии с другими цветами — и смотри-ка, тоже оказались лишними. Короче приделал фичу, а потом отковыриваешь. Но на самом деле это классика, я оч люблю переусложнить и ВСЕ учесть. Чаще всего получается сделать только первое.

Команде не понравилось что демо-страница архитектурно не вписалась в проект. Короче PM завёл отдельный issue с согласованной архитектурой под это.

## Решения

- provideRouter убран из app.spec.ts — RouterOutlet в тестах работает и без него
- CVA выброшен целиком, компонент использует model() для двустороннего связывания — проще и честнее относительно текущих требований
- status: 'default' | 'error' | 'warning' заменён на два boolean-инпута error и warning — API проще, лишний тип удалён
- inputId убран: лейбл внутри компонента, для тестов есть data-testid, crypto.randomUUID() справляется сам
- Лейбл рендерится всегда без внешнего @if — иначе required-звёздочка не показывалась при пустом лейбле
- Демо забэкапировано в отдельную ветку backup/input-demo и вырезано из PR

## Что я узнал

Angular @let в шаблонах — удобный способ не вызывать один сигнал несколько раз, код читается чище. model() покрывает большинство реальных кейсов без накладных расходов CVA. YAGNI работает: CVA, градации цветов, inputId — всё это казалось полезным заранее, но не пригодилось. CI-окружение (Linux) может вести себя иначе чем локальная macOS, особенно в части разрешения TypeScript-идентификаторов.
